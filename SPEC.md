# Mikrokosmos SPEC

## 1. 프로젝트 개요

**Mikrokosmos**는 사용자의 꿈, 목표, 소원을 **별(Star)**로 생성하고, 하루의 기록을 통해 그 별을 변화시키며, 이를 **우주 인터페이스(Cosmos UI)** 위에서 탐색하고 공유하는 웹/앱 서비스이다.

이 서비스는 겉으로는 감성 성장 플랫폼처럼 보이지만, 내부적으로는 인간이 품는 목표와 바람, 그리고 그것이 시간에 따라 유지되거나 사라지는 과정을 축적하는 **Human Aspiration Data Platform**으로 설계된다.

Mikrokosmos의 핵심은 단순한 목표 관리가 아니다. 이 서비스는 인간의 삶을 다음과 같이 해석한다.

* 한 사람은 여러 개의 꿈과 소원을 품을 수 있다. (총 5개의 별을 등록 할수 있음)
* 그 꿈과 소원은 별처럼 우주 공간에 존재한다.
* 기록은 별의 생명과 밝기를 유지하는 행위다. (매일 기록하지 않으면 별이 서서히 사라짐)
* 사라진 꿈도 완전히 없어지지 않고 흔적으로 남는다. (별을 지워도 잔상이 남는 것 )
* 사람들은 각자의 별을 가지고 있지만, 모두 하나의 공통된 우주 안에 존재한다.

즉, **Mikrokosmos는 개인의 성장 인터페이스이면서 동시에 익명 기반의 집단적 꿈 지도다.**

---

## 2. 서비스 철학

### 2.1 핵심 메타포

Mikrokosmos의 설계는 다음 메타포를 중심으로 구성된다.

| 개념          | 메타포     |
| ----------- | ------- |
| 꿈 / 목표 / 소원 | 별       |
| 하루의 기록      | 별의 에너지  |
| 지속성         | 별의 생명력  |
| 삭제된 꿈       | 흔적 별    |
| 유사한 꿈들의 집합  | 은하 / 군집 |
| 전체 사용자 공간   | 하나의 소우주 |

이 서비스는 사용자를 평가하지 않는다. 사용자의 기록은 오직 **자기 신뢰**를 기반으로 별에 반영된다.

이 원칙은 매우 중요하다. Mikrokosmos는 생산성 앱이 아니라, 인간이 자기 삶을 판단 없이 바라보되 구조적으로 시각화하는 공간이어야 한다.

### 2.2 서비스의 본질

Mikrokosmos는 다음 세 가지 층위를 동시에 가진다.

#### 감성적 층위

사용자는 자신의 꿈과 소원을 별로 만들고, 이를 우주처럼 바라보며 감성적으로 몰입한다.

#### 행동적 층위

사용자는 하루 기록을 남기며 별을 유지하고 성장시킨다. 기록이 없으면 별은 쇠퇴한다.

#### 데이터적 층위

플랫폼은 인간이 어떤 꿈을 꾸는지, 어떤 소원이 오래 유지되는지, 어떤 목표가 쉽게 버려지는지 등 **목표 지속성 데이터**를 축적한다.

## 3. 제품 목표

### 3.1 사용자 관점 목표

사용자에게 Mikrokosmos는 다음과 같은 경험을 제공해야 한다.

* 자신의 꿈과 소원을 시각적으로 보고 싶다.
* 하루를 어떤 방향으로 살았는지 감각적으로 확인하고 싶다.
* 사라진 목표조차 삶의 흔적으로 남기고 싶다.
* 비슷한 목표를 가진 다른 사람들의 별을 익명으로 보고 싶다.
* 타임라인 SNS가 아니라, 더 고요하고 구조적인 공간에서 타인의 의지를 관찰하고 싶다.

### 3.2 플랫폼 관점 목표

플랫폼은 다음을 축적하는 구조를 가진다.

* 인간의 목표 텍스트 데이터
* 목표의 유형 데이터
* 목표 간 유사도 데이터
* 목표 유지 기간 데이터
* 목표 생성 / 소멸 패턴 데이터
* 기록 지속성 데이터

즉, Mikrokosmos는 감성 성장 앱처럼 보이지만, 실제로는 **인간 목표와 지속성에 관한 구조화된 데이터 레이어를 만드는 서비스**다.

---

## 4. 사용자 경험 정의

### 4.1 서비스의 기본 경험

사용자는 Mikrokosmos에 들어와 다음 흐름을 경험한다.

1. 자신의 별을 생성한다.
2. 별에 초기 문장을 남긴다.
3. 별은 즉시 우주 공간에 배치된다.
4. 사용자는 별에 하루 기록을 쌓는다.
5. 별은 기록에 따라 밝기와 상태가 변화한다.
6. 다른 사람의 별을 탐색할 수 있다.
7. 삭제한 별도 흔적으로 우주에 남는다.

### 4.2 핵심 감정 경험

이 서비스는 사용자가 다음 감정을 느끼도록 설계한다.

* “내 꿈이 실제로 우주에 떠 있다.”
* “오늘 내가 이 별을 조금 더 밝게 만들었다.”
* “예전에 품었던 마음도 완전히 사라지지 않았다.”
* “세상에는 나와 비슷한 별을 가진 사람들이 많다.”
* “나는 혼자가 아니라, 하나의 소우주 안에 있다.”

---

## 5. 서비스 구조

### 5.1 세계 구조

Mikrokosmos는 기본적으로 하나의 공통 우주(**shared universe**) 위에서 동작한다.

각 사용자는 자신의 별들을 생성하지만, 이 별들은 전부 동일한 우주 안에 배치된다. 즉, 이 서비스는 개인 앱이 아니라, **개인 중심이지만 타인의 별도 보이는 익명 기반 우주형 플랫폼**이다.

### 5.2 익명성 원칙

사용자는 다른 사용자의 실명이나 상세 정체성을 보지 않는다. 핵심 객체는 사람이 아니라 **별**이다.

플랫폼 상에서 중요한 것은 **“누가 썼는가”**보다 **“어떤 별이 존재하는가”**이다.

* 사용자 정체성은 최소화한다.
* 닉네임은 선택적이거나 최소 노출한다.
* 프로필 중심 구조를 지양한다.
* 콘텐츠 주체는 사용자보다 별이다.
* 탐색 단위는 사람보다 별과 군집이다.

---

## 6. 핵심 개념 모델

### 6.1 사용자 (User)

사용자는 여러 개의 별을 만들 수 있다. 단, 한 사용자가 생성할 수 있는 별 수는 **최대 5개**다.

이 제한의 이유는 다음과 같다.

* 너무 많은 별 생성으로 인한 우주 오염 방지
* 각 별에 대한 의미 밀도 유지
* 목표 남발 대신 선택적 생성 유도

### 6.2 별 (Star)

별은 사용자가 품고 있는 꿈, 목표, 소원, 바람을 나타내는 핵심 객체다.

별은 다음 특징을 가진다.

* 하나의 테마를 가진다.
* 하나의 초기 문장을 가진다.
* 여러 개의 기록을 가질 수 있다.
* 시간이 지나며 상태가 변화한다.
* 삭제되어도 완전히 사라지지 않는다.
* 유사한 별들과 가까운 위치에 배치된다.
* 생성 즉시 카탈로그 기반 이름을 부여받는다.

### 6.3 기록 (Log)

기록은 사용자가 별에 남기는 하루 단위 텍스트다. Mikrokosmos에서는 기록의 의미를 AI가 평가하지 않는다.

기록은 다음 의미를 가진다.

* 별을 유지하는 행위
* 별의 생명력을 증명하는 행위
* 사용자의 하루를 특정 꿈과 연결하는 행위

기록의 존재 자체가 중요하지, 기록의 질을 시스템이 심사하지 않는다.

### 6.4 군집 (Galaxy / Cluster)

유사한 별들은 벡터 공간상에서 가까운 위치에 배치된다. 이때 은하는 “예술”, “기술” 같은 일반 카테고리보다는, 서비스 내부에서 정의한 상위 테마와 실제 유사성 기반 배치를 함께 사용한다.

즉 군집은 다음 두 축을 가진다.

* 상위 테마 분류
* 의미 유사도 기반 근접 배치

---

## 7. 별의 테마 구조

### 7.1 기본 방향

별은 지나치게 세분화된 카테고리로 시작하지 않는다. 초기 버전에서는 사람의 바람을 담는 상위 개념 중심으로 시작한다.

#### 예시 테마

* 꿈
* 목표
* 소원
* 바람

초기 MVP에서는 너무 많은 테마를 두지 않는다. 핵심은 분류의 정밀함보다, 사용자가 직관적으로 자신의 별을 어디에 둘지 선택할 수 있게 하는 것이다.

### 7.2 테마별 색상 예시

테마에 따라 별의 색상은 다르게 표현된다.

| 테마 | 색상 예시         |
| -- | ------------- |
| 꿈  | 노란색 계열        |
| 목표 | 흰색 또는 금색 계열   |
| 소원 | 파란색 계열        |
| 바람 | 보라색 또는 청록색 계열 |

색상 체계는 이후 브랜딩과 함께 정교화하되, 초기에는 시각적 구분과 감성적 몰입을 동시에 만족해야 한다.

## 8. 별 이름 시스템

### 8.1 목적

별 이름은 Mikrokosmos 세계관을 강화하는 핵심 요소다. 일반적인 게시물 ID가 아니라, **천문 카탈로그형 명명 방식**을 사용한다.

사용자가 별을 생성하면 즉시 별 이름이 부여된다.

### 8.2 명명 원칙

별 이름은 실제 천문 카탈로그 스타일을 참고한다.

#### 예시 형식

* HD 189733
* HIP 94779
* GJ 1214
* 2MASS J19233820+4816046

단, 초기 MVP에서는 지나치게 긴 형식은 피하고, 사용자 경험을 해치지 않는 선에서 간결한 체계를 우선 적용한다.

#### 간결형 예시

* HD 284921
* HIP 48391
* GJ 2048

### 8.3 표시 구조

별은 두 개의 이름을 가진다.

#### 시스템 이름

카탈로그 기반 자동 부여 이름

#### 사용자 의미 이름

사용자가 작성한 제목 또는 초기 의미 텍스트

#### 예시

* 시스템 이름: HD 284921
* 의미 이름: AI 연구자가 되고 싶다

사용자 UI에서는 이 둘을 함께 보여줄 수 있다.

## 9. 별 상태 시스템

별은 단순히 생성되고 끝나는 객체가 아니다. 시간과 기록 여부에 따라 상태가 변화한다.

초기 상태 모델은 다음 3가지를 기준으로 한다.

* 활성 별
* 죽어가는 별
* 흔적 별

### 9.1 활성 별 (Active Star)

현재 기록이 지속되고 있는 별이다.

#### 특징

* 밝기가 선명하다.
* 생명력이 유지된다.
* 우주 상에서 가장 살아 있는 존재처럼 보인다.
* 최근 활동이 존재한다.

#### 시각적 연출 방향

* 부드러운 발광
* 미세한 반짝임
* 선명한 테마 색상
* 주변 glow 효과

### 9.2 죽어가는 별 (Dying Star)

일정 기간 기록이 없는 별이다.

#### 특징

* 아직 삭제되지는 않았지만 생명력이 약해진 상태
* 사용자가 다시 기록하면 회복 가능
* 방치된 꿈 / 멈춘 목표를 상징

#### 시각적 연출 방향

* 빛이 약해짐
* 불규칙한 깜빡임
* 색이 탈색되거나 붉어짐
* glow 감소

이 상태는 사용자에게 죄책감을 주기보다는, **“이 별은 아직 완전히 사라지지 않았다”**는 감각을 줘야 한다.

### 9.3 흔적 별 (Archived Star)

사용자가 직접 삭제한 별이다. 하지만 Mikrokosmos에서는 삭제가 곧 완전 소멸을 의미하지 않는다.

삭제된 별은 초심만 남긴 채 우주에 잔광처럼 남는다.

#### 특징

* 전체 기록은 보이지 않는다.
* 생성 당시의 초기 문장만 보여준다.
* 더 이상 성장하지 않는다.
* 완전히 사라진 것이 아니라, “한때 품었던 마음의 흔적”으로 존재한다.

#### 시각적 연출 방향

* 희미한 점 또는 성운형 잔광
* 거의 정적
* 낮은 밝기
* 먼지 / 입자 효과
* 탈색된 컬러

흔적 별은 이 서비스의 철학적 핵심 중 하나다. Mikrokosmos는 버려진 꿈조차 제거하지 않고 삶의 흔적으로 남긴다.

## 10. 별 생명주기

### 10.1 생명주기 개요

별은 다음 흐름을 가진다.

1. 생성
2. 기록 축적
3. 성장
4. 유지 또는 쇠퇴
5. 방치되면 죽어감
6. 삭제되면 흔적으로 전환

### 10.2 성장 로직의 방향

Mikrokosmos는 사용자의 자기 신뢰 기반 서비스이므로, 성장 로직은 최대한 단순하고 상징적이어야 한다.

AI 분석 없이, 기록의 축적에 따라 별의 단계가 올라간다.

#### 예시 개념

* 초기 생성
* 작은 별
* 안정된 별
* 더 밝은 별
* 큰 별

초기 MVP에서는 실제 천문학 용어를 일부 차용할 수 있다.

#### 예시 단계

* Protostar
* Young Star
* Main Sequence
* Giant
* Supergiant

단, 사용자 UX에서 너무 과학 용어에 치우치지 않도록 밸런스를 잡아야 한다.

### 10.3 죽어가는 기준

기록 부재로 인해 활성 별이 죽어가는 별로 전환되는 기준은 제품 설계 단계에서 정한다.

#### 예시 기준

* N일 이상 기록 없음 → Dying
* 더 장기 기록 없음 → 빛 더 약화

초기 버전에서는 단순한 시간 기준을 우선 적용한다.

## 11. 삭제 정책

### 11.1 삭제의 의미

삭제는 DB에서 완전히 제거하는 기능이 아니다. 삭제는 사용자 관점에서는 **“이제 이 별을 더 키우지 않겠다”**는 의미다.

시스템적으로는 별을 **ARCHIVED 상태**로 전환한다.

### 11.2 흔적 별에 표시되는 정보

흔적 별에서는 전체 기록을 보여주지 않는다. 초기 문장만 보여준다.

#### 표시 예시

* 시스템 이름
* 테마
* 생성 시점
* 첫 문장
* 현재는 기록 중단 상태라는 의미 정보

#### 표시하지 않는 것

* 전체 로그
* 상세 성장 히스토리
* 최근 활동
* 내부 점수

즉 흔적 별은 활동 객체가 아니라 **기억 객체**다.

---

## 12. 우주 구조와 배치 로직

### 12.1 공통 우주

모든 별은 하나의 공통된 소우주 안에 존재한다. 사용자는 자신의 별만 보는 것이 아니라, 다른 사람의 별도 탐색할 수 있다.

### 12.2 배치 원칙

별은 무작위로 뿌려지지 않는다. 별의 위치는 의미 공간 상에서 결정된다.

#### 기본 원칙

* 유사한 꿈 / 목표 / 소원은 가까운 위치
* 상이한 별은 먼 위치
* 동일 테마끼리는 시각적으로 군집 형성 가능
* 타 카테고리 별로 이동도 가능

즉 Mikrokosmos의 우주는 **의미 기반 지도**다.

### 12.3 벡터 공간 사용 원칙

플랫폼 내부에서는 별의 텍스트를 기반으로 벡터 공간에 배치한다. 이는 사용자에게 직접 드러나는 기술 요소는 아니지만, 우주 인터페이스의 자연스러운 공간 구조를 위한 핵심 로직이다.

초기 단계에서는 다음 정도만 정의한다.

* 별은 의미 기반 임베딩을 가진다.
* 벡터 유사도에 따라 위치가 결정된다.
* 유사한 별들은 모여 하나의 영역감을 형성한다.

---

## 13. 탐색 (Explore) 경험

### 13.1 탐색의 목적

Mikrokosmos에서 타인의 별을 보는 이유는 “남을 구경하기 위해서”가 아니다.

핵심은 다음과 같다.

* 나와 비슷한 바람을 가진 사람을 간접적으로 발견
* 세상에 어떤 별들이 존재하는지 관찰
* 인간의 다양한 의지를 시각적으로 체감
* 익명의 별들을 통해 공감과 동기 형성

### 13.2 탐색 방식

초기 탐색 방식 예시:

* 유사한 별 보기
* 같은 테마 별 보기
* 랜덤 별 탐색
* 다른 영역으로 이동하기

탐색은 사람 프로필이 아니라 **별 중심 구조**로 동작한다.

---

## 14. 콘텐츠 노출 원칙

### 14.1 활성 / 죽어가는 별

활성 별과 죽어가는 별은 사용자가 남긴 콘텐츠를 볼 수 있다.

단, 공개 범위와 익명성은 서비스 정책에 따라 조정 가능하다.

### 14.2 흔적 별

흔적 별은 **초기 문장만 노출**한다.

이 원칙은 매우 중요하다. 삭제된 별이 모든 히스토리를 노출하면 “삭제”의 의미가 사라진다. 반대로 완전히 제거하면 Mikrokosmos 세계관이 약해진다.

그래서 흔적 별은 **처음의 마음만 남기는 구조**로 유지한다.

## 15. UI / UX 방향

### 15.1 UI 핵심 방향

UI는 단순한 카드 나열형 SNS가 아니라, **우주를 탐험하는 인터페이스**여야 한다.

#### 핵심 레퍼런스

* Google Earth 스타일의 줌인 / 줌아웃 경험
* 3D처럼 느껴지는 공간감
* 실제 우주를 떠다니는 듯한 인터랙션
* 별을 중심으로 탐색이 이어지는 구조

### 15.2 인터랙션 원칙

사용자는 다음 행동을 자연스럽게 할 수 있어야 한다.

* 우주를 이동한다.
* 특정 군집으로 확대한다.
* 별을 클릭한다.
* 별의 내용과 기록을 본다.
* 자신의 별로 이동한다.
* 다른 테마 영역으로 이동한다.

### 15.3 UI 감성 기준

Mikrokosmos는 감성적이어야 하지만 싸구려 감성 앱처럼 보이면 안 된다.

#### 원칙

* 미니멀할 것
* 과한 텍스트 장식 지양
* 세계관은 강하지만 연출은 절제할 것
* 별 상태는 직관적으로 다르게 보일 것
* 서비스 전체가 조용하고 깊은 공간처럼 느껴질 것

## 16. 별 상태별 UI 가이드

### 16.1 활성 별

**표현 목표:** 살아 있는 꿈

#### 권장 표현

* 선명한 색상
* pulse glow
* 미세한 shimmering
* 안정적인 광원
* 우주 상에서 중심감 있는 시각 존재

사용자는 활성 별을 보고 **“지금도 계속 살아 있는 목표”**라는 느낌을 받아야 한다.

### 16.2 죽어가는 별

**표현 목표:** 멈춰 있지만 아직 사라지지 않은 꿈

#### 권장 표현

* 빛의 세기 감소
* 불규칙 flicker
* glow 약화
* 색상 탈색 또는 불안정한 붉은 기운
* 주변 입자 감소

사용자는 죽어가는 별을 보고 **“다시 살릴 수 있다”**는 감각을 가져야 한다.

### 16.3 흔적 별

**표현 목표:** 지나간 마음의 흔적

#### 권장 표현

* 매우 약한 밝기
* 성운형 잔광 또는 미세한 점
* 거의 정적인 형태
* 먼지 / 입자 효과
* 탈색된 계열

사용자는 흔적 별을 보고 **“한때 내가 품었던 별”**이라는 감정을 느껴야 한다.

## 17. 사용자 행동 제한 및 정책

* 삭제는 자유롭게 가능

## 20. 제품 포지셔닝

Mikrokosmos는 다음과 같은 서비스들의 중간이 아니다. 오히려 이들의 빈 공간에 위치한다.

* 일반 투두 / 목표 관리 앱과 다름
* 일기 앱과 다름
* 타임라인형 SNS와 다름
* 감정 기록 앱과도 다름

Mikrokosmos는 다음으로 정의된다.

> 우주형 인터페이스를 가진 익명 기반 자기성장 및 목표 흔적 플랫폼

또는 더 직설적으로 말하면,

> 꿈을 별로 만들고, 기록으로 빛나게 하며, 사라진 마음조차 흔적으로 남기는 서비스

## 21. 차별점

### 21.1 기존 목표 관리 앱과 차이

기존 서비스는 체크리스트 중심이다. Mikrokosmos는 **존재와 지속성**을 시각화한다.

### 21.2 기존 SNS와 차이

기존 SNS는 사람과 피드 중심이다. Mikrokosmos는 **별과 우주 중심**이다.

### 21.3 기존 일기 앱과 차이

기존 일기 앱은 과거 회고에 가깝다. Mikrokosmos는 **미래지향적 별**을 중심에 둔다.

### 21.4 삭제 철학의 차이

기존 서비스는 삭제 = 제거다. Mikrokosmos는 **삭제 = 흔적화**다.

## 24. 제품 설계 원칙 요약

Mikrokosmos는 다음 원칙을 반드시 유지해야 한다.

* 기록은 자기 신뢰를 기반으로 한다.
* 별은 단순 게시물이 아니라 삶의 방향성이다.
* 삭제된 별도 우주에 흔적으로 남는다.
* 사람보다 별이 중심 객체다.
* 우주는 단순 배경이 아니라 탐색 가능한 인터페이스다.
* 감성은 중요하지만 구조가 먼저다.
* 이 서비스는 겉으로는 성장 앱, 내부적으로는 인간 목표 데이터 플랫폼이다.
